Before learning commands, it helps to separate the terminal from the programs that run inside it.

## The terminal and the shell are related but not identical

In everyday usage, people often say "terminal" for the whole experience.

More precisely:

- a **terminal app** gives you the text-based window
- a **shell** reads the commands you type and runs them

Examples of shells include:

- `zsh`
- `bash`
- `fish`

You do not need to master shell differences first. But it helps to know that the terminal window and the command interpreter are not exactly the same thing.

## Why developers use the terminal so often

Many developer tools are designed to be run as commands.

That has practical advantages:

- commands are fast to repeat
- commands can be written in documentation exactly
- commands work well in automation
- commands can be combined into workflows later

That is why the terminal keeps showing up even when a tool also has a graphical interface.

## The terminal is especially useful for structured work

The terminal is good at tasks such as:

- moving around a project directory
- starting a development server
- installing dependencies
- running tests
- inspecting files
- executing automation and scripts

It is not always the best interface for every task. But it is often the clearest interface for developer tooling.

## A useful beginner mindset

The terminal is not mainly about memorizing commands.

It is about understanding a small model:

- where you are
- what files are here
- what command you want to run
- what arguments should be passed to it

That model is more important than any single command.
